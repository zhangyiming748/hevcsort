# HEVC Sort 项目逻辑漏洞与改进建议

## 🔴 严重问题

### 1. 跨分区文件移动失败

**位置**: `core/core.go` 第 65 行

**问题描述**:
```go
if err := os.Rename(video, target); err != nil {
    log.Printf("移动文件 %s 到 %s 失败: %v", video, target, err)
    continue
}
```


- `os.Rename` 在 Windows 上不能跨驱动器/分区移动文件
- 如果 src 和 dst 在不同磁盘分区（如 C: 和 D:），会直接失败
- Linux/macOS 上同文件系统可以，但跨挂载点也会失败

**解决方案**:

如果main函数接收到的的src dst 不在同一个分区上 比如 c:\Users d:\data 或 /sda1 /sdb2 直接报一个警告然后退出
Windows linux macos 一视同仁 跨分区就算失败
只在输入参数的时候判断一次即可 后边不用改

### 2. 缺少目录存在性检查

**位置**: `main.go` PreRun 和 `core/core.go` Split 函数

**问题描述**:
- PreRun 中只检查了根目录下是否有视频文件，但没有检查目录本身是否存在
- 如果 src 或 dst 目录不存在，`finder.FindAllVideosInRoot` 可能返回空列表而不是报错
- 导致"未找到文件"的误导性输出，实际是目录不存在

**解决方案**:
- [ ] 在 PreRun 中添加 `os.Stat` 检查 src 和 dst 是否存在
- [ ] 验证 src 和 dst 是目录而非文件
- [ ] 在 Split 函数开始时也进行同样的检查
- [ ] 如果目录不存在，给出明确错误提示

---

### 3. PreRun 错误处理方式不当

**位置**: `main.go` 第 32、36、41 行

**问题描述**:
```go
log.Fatalf("源目录和目标目录不能相同...")
log.Fatalf("❌ 错误: src 根目录下不应该有视频文件...")
log.Fatalf("❌ 错误: dst 根目录下不应该有视频文件...")
```

- 根据 Cobra 最佳实践，PreRun 中应该使用 `os.Exit(1)` 明确终止
- `log.Fatalf` 虽然会调用 `os.Exit(1)`，但不够显式
- 更好的做法是使用 `fmt.Fprintf(os.Stderr, ...)` + `os.Exit(1)`

**解决方案**:
- [ ] 将 `log.Fatalf` 改为 `fmt.Fprintf(os.Stderr, ...)` + `os.Exit(1)`
- [ ] 确保错误信息清晰明了
- [ ] 保持错误码为 1 表示失败

## 🟡 中等问题

### 4. 没有处理文件冲突

**位置**: `core/core.go` 第 65 行

**问题描述**:
- 如果目标位置已存在同名文件，`os.Rename` 会失败
- 用户无法选择覆盖、跳过或重命名策略
- 可能导致部分文件处理失败，用户不知道原因

**解决方案**:
- [ ] 或者自动重命名（如添加时间戳或序号）
- [ ] 在日志中明确记录冲突处理方式 原文件名 避免重复而添加时间戳的新文件名

---

### 5. 测试代码硬编码路径

**位置**: `core/move_test.go` 第 9-10 行

**问题描述**:
```go
src := "G:\\archive\\src"
dst := "G:\\archive\\dst"
```

- 硬编码了特定路径，不具备可移植性
- 其他开发者运行测试时会失败
- CI/CD 环境中路径肯定不存在
- 根据记忆经验，这个路径可能不存在导致测试失败

**解决方案**:
- [ ] 使用 `testing.T.TempDir()` 创建临时测试目录
- [ ] 创建模拟的视频文件进行测试
- [ ] 或者将测试标记为 integration test，需要手动启用
- [ ] 添加跳过逻辑：如果路径不存在则跳过测试

---

### 6. 缺少用户反馈和进度显示

**位置**: `core/core.go` 第 35-71 行的循环

**问题描述**:
- 处理大量文件时（如数千个视频），用户看不到进度
- 只能看到最终的"分类完成！"
- 长时间无响应会让用户以为程序卡死

**解决方案**:
- [ ] 添加进度条显示（如使用 `progressbar` 库）
- [ ] 或至少显示当前处理进度：`正在处理 123/1000...`
- [ ] 显示预计剩余时间（可选）
- [ ] 每处理 N 个文件输出一次状态

---

## 🟢 轻微问题

### 7. 日志输出不一致

**位置**: `core/core.go` 多处

**问题描述**:
- 第 20、35、73 行使用 `fmt.Println/Printf`
- 第 53、60、66、70 行使用 `log.Printf`
- 成功和失败消息混用不同的输出方式
- `log` 包默认会添加时间戳前缀，而 `fmt` 不会

**解决方案**:
统一使用 `log` 包并配置格式

---

### 8. 没有并发控制

**位置**: `core/core.go` 第 37-71 行的 for 循环

**问题描述**:
- 串行处理所有视频文件
- 每个文件都需要调用 `FastMediaInfo.GetStandMediaInfo`，这是 I/O 密集型操作
- 可以利用多核 CPU 并行处理提升性能

**解决方案**:
为确保兼容性能羸弱的设备 不要使用任何多线程策略
